# PrimaPrompter v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship v1 of PrimaPrompter — a free, open-source, on-device, vertical-first teleprompter camera app for iOS + Android from a single React Native codebase — against the approved 12-screen wireframes, published to both stores.

**Architecture:** React Native app; `react-native-vision-camera` v5 owns capture; the teleprompter is plain RN views composited over the `<Camera>` preview, scrolled on the UI thread via Reanimated. All persistence is on-device (op-sqlite for relational data, MMKV for prefs). Stack-based navigation anchored on the Script Library. No backend, no accounts, no network.

**Tech Stack:** React Native (Expo dev-build — config plugin + expo-dev-client — provisional, spike-confirmed), react-native-vision-camera@5, react-native-reanimated, react-native-gesture-handler, @react-navigation/native (native-stack), op-sqlite, react-native-mmkv, TypeScript, Jest + React Native Testing Library. **Node 22 LTS** (Node 25 is non-LTS and unsupported by Expo).

**Source of truth:** design spec `docs/superpowers/specs/2026-07-10-primaprompter-design.md`; approved wireframes (artifact, 12 screens). Reviewed by product-brain (approve-with-amendments, all folded in).

## Global Constraints

Copied verbatim from the spec — every task's requirements implicitly include these:

- **On-device only** — no network calls, no backend, no accounts, no analytics/telemetry. The only data egress is a user-initiated share.
- **Free + open source**, public repo, published to both stores. **Apache-2.0** LICENSE must land before the repo is made public.
- **Vertical-first 9:16**; portrait-locked in v1.
- **Sole git author:** Mayank Gupta `<techfreakworm@gmail.com>` — no `Co-Authored-By`, no "Generated with" footers.
- **Camera controls are capability-gated** — query device formats **per lens**; only offer what the device reports; grey out impossible combos (incl. the codec×HDR×fps dependency matrix) with a plain-language reason. Never promise "manual exposure" (it is tap-to-focus + AE/AF lock + exposure compensation).
- **Prompter scroll runs on the UI thread** (Reanimated native driver) — never JS-thread during encode.
- **Never lose a take** — call / backgrounding / storage-full / thermal / BT-mic route-change mid-take must finalize and save the take.
- **Accessibility is day-one** — shared components carry minimum touch targets + label props as a construction standard; the audit is Phase 9, not the introduction.
- **v1 hard requirements (non-screen):** keep-awake during record; storage guard (estimate MB/min vs free space, warn before a take that won't fit); portrait lock.
- **Cue tags are v1-passive** — color only; excluded from word/duration math; trigger no behavior.
- **Working name** "PrimaPrompter"; final name / icon / bundle IDs / dev-accounts are owner-TBD before submission (Phase 10).
- **TDD** throughout: failing test → run (fail) → minimal impl → run (pass) → commit. Frequent commits.

## Phase 1 — Device spike (Milestone 0; de-risk; owner/devops runs on real hardware)

**Deliverable:** a written verdict in `docs/superpowers/architecture/spike-results.md` resolving every gating unknown, each gate carrying a "what happens on each failure" line. Each probe extends the Phase-0 skeleton and runs on **both a modern and a low-end Android plus an iPhone**. Exploratory measurement, **not** TDD.

- [ ] **1.1 UI-thread scroll under encode (+ thermal):** overlay a Reanimated-driven scrolling band; start a 4K/60 recording (where supported); measure dropped frames on the low-end Android. Run one **long 4K record** here for best-effort thermal observation. **Gates:** the UI-thread-scroll mandate (fail → cap fps or revisit). Record fps + smoothness + any thermal throttle.
- [ ] **1.2 Format query + gating matrix (PER LENS):** enumerate `device.formats` for **both front and back** — front is the default and usually weaker, so a back-only matrix misgates the majority path. Build the offered resolution/fps/codec/HDR/stabilization set; confirm **H.264+HDR is correctly excluded** and cinematic-stabilization is iOS-only. **Gates:** Capture Settings gating logic (P5's core input; fail → widen/narrow the offered set). Record the matrix per lens per device.
- [ ] **1.3 Interruption-finalize (+ error taxonomy):** during recording trigger incoming call, backgrounding, simulated storage-full, and a **BT-mic route change mid-take** (min bar: no crash, take survives); confirm each finalizes to a playable take. Document the SDK's interruption/error taxonomy — thermal shutdown shares this finalize-on-error path. **Gates:** never-lose-a-take (fail → the app's most unforgivable bug; block build until solved). Record behavior per trigger.
- [ ] **1.4 Passthrough trim (timeboxed probe → estimate):** timebox the **probe** to ~1 day; probe single-clip in/out trim with no re-encode + the one re-encode fallback, on **Android AND a 30-min iOS clip** (ships on both). Output is an implementation **estimate**, judged against the ~3–4 dev-day drop threshold (probe-timebox ≠ impl-estimate). **Gates:** Trim-Lite in v1 vs auto-drop to v1.1. Record feasibility + effort estimate per platform.
- [ ] **1.5 Record-time mirroring:** confirm whether the SDK can write a horizontally-mirrored file at record time (no re-encode). **Gates:** flip-saved-video in v1 (P5) vs v2. Record support per platform.
- [ ] **1.6 iOS volume-button capture:** verify `AVCaptureEventInteraction` availability vs our iOS floor. **Gates:** hardware start/stop on iOS (Android already v1). Record the minimum iOS version.
- [ ] **1.7 AE/AF-lock + control smoke:** the approved wireframe promises "Tap to lock focus & exposure" — verify vision-camera v5 exposes persistent **lock** semantics (tap-focus and exposure-comp are known; a persistent lock is a different API surface). Smoke torch + zoom too. **Gates:** whether AE/AF-lock ships as built, needs a native patch, or the UI copy changes. Record the API reality.
- [ ] **1.8 MB/min measurement:** record short clips per key format (1080p30, 1080p60, 4K30, 4K60; H.264 vs HEVC) and derive real **MB/min per platform** — the storage-guard constants (P2.5) are folklore until measured. **Gates:** storage-guard accuracy. Record the numbers.
- [ ] **1.9 Expo-dev-build vs bare RN:** consolidate 1.1–1.8 friction. **Decision:** confirm Expo dev-build, or migrate to bare RN (flips the toolchain every later phase assumes). Record the call + rationale.
- [ ] **1.10 Write the verdict** to `spike-results.md`, one "on failure →" line per gate; update the spec's §11/§12 if any gate flips; commit.

> **Owner checkpoint:** the spike verdict is reviewed (owner + product-brain) before feature build begins. Any stack reopen or scope change is decided here.

## Phase 2 — Foundation (post-spike; TDD; freezes the interfaces Phases 3–10 consume)

**Deliverable:** design tokens, navigation shell, the on-device data layer, shared components, and repo hygiene. All pure-logic units are TDD. **This phase's job is interface-freezing — hold it hard to that.**

- **2.1 Design tokens** (`src/theme/tokens.ts`): port the wireframe palette (tally amber `#E8930A`, stage/paper grounds, cue cyan, record red) and type roles (display/UI/mono); the two-world (dark camera / light studio) theming primitive.
- **2.2 Navigation shell** (`src/navigation/`): native-stack, Library home + Settings; typed routes; record/review/trim pushed stack.
- **2.3 Data layer** (`src/db/`): op-sqlite schema + migrations for `Folder`, `Script` (incl. `wpmOverride`), `Take` (incl. `trimmedFromTakeId`); repositories with typed CRUD. **TDD** each repository against a temp DB. Migrations are **shape-gated repairs, not version-gated** (house rule).
- **2.4 Prefs store** (`src/store/prefs.ts`): MMKV-backed `TeleprompterPrefs`, `CaptureSettings`, `diagnosticsOptIn`. TDD getters/setters/defaults.
- **2.5 Core lib** (`src/lib/`): `estimateDuration(wordCount, wpm)` (TDD — cue tags excluded), `storage` guard (TDD the MB/min-vs-free math using **spike 1.8's measured constants**), `diagnostics` ring buffer (TDD append/cap/serialize), `keepAwake` wrapper.
- **2.6 Accessibility construction standards:** shared components (`src/components/`) bake in minimum touch targets + required accessibility label props from the start — day-one, not a P9 bolt-on. (P9 is the audit, not the introduction.)
- **2.7 Repo hygiene:** `LICENSE` (Apache-2.0), `README` (privacy/OSS positioning), `CONTRIBUTING`, `CODE_OF_CONDUCT`, `SECURITY.md`, issue/PR templates, `.github/workflows/ci.yml` (typecheck + lint + test; no secrets). **LICENSE lands before any public push.**

## Phases 3–10 — Feature build + release (post-spike; TDD; each maps to approved wireframes)

Each phase produces one working, testable slice. **Each per-phase detailed plan passes this same review gate (owner-lens) before its execution** — cheap, since scope is already fenced. Sequenced by dependency:

- **Phase 3 — Script Library + Editor** (wireframes 01–03): folder/script list, search, CRUD, guided empty-state incl. **"Paste from clipboard"** CTA; **`.txt` document-picker import** (one-shot OS picker, not cloud sync); editor with passive color-coded cue tags, live word-count + duration. TDD: repositories wiring, duration display, cue-tag rendering (passive), import parse.
- **Phase 4A — Record spine + reliability** (wireframes 04–05, camera half): camera + shutter; the **state machine** (idle / countdown / recording / finalizing) — the finalize-on-interruption transitions ARE the state machine, kept together; keep-awake; portrait lock; storage guard; hardware start/stop (Android; iOS per 1.6). TDD: state-machine transitions, interruption→finalize matrix (a test case per trigger from 1.3), storage-guard math (constants from 1.8).
- **Phase 4B — Teleprompter overlay + live controls** (wireframes 04–05, overlay half): pinned high band + fixed caret on the active line; UI-thread WPM scroll; live WPM rail (persists dimmed while recording); tap-to-pause, drag-scrub; per-script speed; collapse-on-record chrome. **Camera-independent — buildable against a mock preview, so 4A/4B parallelize under subagents; integration point = the countdown arming both.** TDD: WPM→scroll-rate math, per-script speed persistence, scroll state (play/pause/scrub).
- **Phase 5 — Capture Settings** (wireframe 06): capability-gated sheet from `useCameraFormats` (matrix from 1.2, per-lens); res/fps/codec/stabilization/HDR/exposure-comp + **AE/AF-lock as built per 1.7**/torch/zoom/grid/countdown; **flip-saved-video ships here iff 1.5 passed**, else deferred; greyed combos with reasons. TDD: the gating function (device formats → offered options, incl. codec×HDR×fps exclusions).
- **Phase 6 — Take Review + Trim-Lite** (wireframes 07, 08, 09): auto-review on stop (keep/retake/discard, multi-take); **non-destructive Trim-Lite** (new take + `trimmedFromTakeId` + delete-original) **only if 1.4 cleared it within the drop threshold**, else defer to v1.1; Prompter Appearance sheet (fonts incl. dyslexia, size/line-height/opacity, high-contrast, mirror-off-default). TDD: trim → new take + lineage; keeper/discard mutations.
- **Phase 7 — Takes Gallery** (wireframe 10): takes grouped by script, res/fps badges, keeper stars, save-to-Photos/share, free-space footer. TDD: grouping + free-space display.
- **Phase 8 — Settings + Diagnostics** (wireframe 11): storage management, privacy block, opt-in "Share diagnostics" (ring-buffer log, user-share-only egress), repo/license/version. TDD: diagnostics opt-in gate (nothing leaves without the share action).
- **Phase 9 — Permission priming + accessibility AUDIT** (wireframe 12): first-run permission-prime modal + guided empty-state; then a full **audit** — VoiceOver/TalkBack, Dynamic Type, contrast, touch targets across all screens (components already carry the P2.6 day-one standards; this verifies + fills gaps).
- **Phase 10 — Release (owner-gated)** (spec goal: "published to both stores"): brand finalization (**O3 name / icon / bundle IDs — prerequisite**); Apple privacy manifest + Play Data Safety filed to match the diagnostics stance actually built; privacy-forward store listings; **O5 soak** (1–2 wk TestFlight / internal track); store submission. **Submission + dev-account/fee decisions are owner-gated.**

**Deferred to v2+ (not in this plan):** export presets, voice-follow (on-device ASR), teleprompter-only mode (v1.1), Watch/second-device remote, bionic/RTL, cloud sync, behavior-triggering cues, landscape.
